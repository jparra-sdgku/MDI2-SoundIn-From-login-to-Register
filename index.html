1MaterialTheme Theory

Understanding how Material Design 3 theming works in Compose

Material Design 3 (Material You) is Google's latest design system for Android. In Jetpack Compose, the entire visual identity of an app — colors, typography and shapes — is configured through MaterialTheme. Every composable nested inside MaterialTheme can access these design tokens.

The three pillars of MaterialTheme

PillarWhat it controlsAccessed via
colorSchemeAll colors in the appMaterialTheme.colorScheme.*
typographyAll text stylesMaterialTheme.typography.*
shapesCorner radii of componentsMaterialTheme.shapes.*

Color roles used in SoundIn

RoleUsage in SoundIn
primaryTopAppBar background, Button background
onPrimaryText and icons placed on top of primary-colored surfaces
surfaceCard and dialog backgrounds
onSurfacePrimary text on white/surface backgrounds
onSurfaceVariantSecondary/supporting text — less prominent than onSurface
errorValidation error messages and destructive action icons
backgroundDefault screen background color
💡 Naming Convention

on[Color] is always readable against [Color]. For example, onPrimary is always legible on a primary background. Never use onPrimary on a white background — it will produce wrong contrast.

2Theme Setup

Configuring the SoundIn ColorScheme in Theme.kt with dynamicColor disabled

In this chapter we configure the SoundIn theme. The key decision is setting dynamicColor = false so the app always shows SoundIn's brand purple regardless of the user's wallpaper on Android 12+.

Configure Color.kt

Define the brand colors as Compose Color objects. These are referenced from Theme.kt when building the LightColorScheme and DarkColorScheme.

File location: ui/theme/Color.kt — this file already exists when you create a new Compose project. Replace its contents with the SoundIn brand colors.
ui/theme/Color.kt
package com.example.soundinp1.ui.theme

import androidx.compose.ui.graphics.Color

// SoundIn brand colors
val SoundInPurple = Color(0xFF6B45C2)       // primary — TopAppBar, Buttons
val SoundInPurpleDark = Color(0xFF4A2D99)   // primaryContainer
val SoundInPink = Color(0xFFE91E63)         // accent for highlights
Color values explained
0xFF6B45C2Hexadecimal color value. The 0xFF prefix means full opacity (alpha = 255). 6B45C2 is the RGB hex for SoundIn purple.
Color()Compose color constructor that accepts an ARGB integer. Android colors are stored as 32-bit integers where the first byte is alpha and the remaining three are RGB.

Configure Theme.kt — set dynamicColor = false

The critical change is setting dynamicColor = false. Without it, Android 12+ devices would override SoundIn's colors with colors derived from the user's wallpaper.

File location: ui/theme/Theme.kt — this file already exists. Find the dynamicColor parameter and change its default value from true to false.
ui/theme/Theme.kt — key change
@Composable
fun SoundInTheme(
    darkTheme: Boolean = isSystemInDarkTheme(),
    dynamicColor: Boolean = false, // ← CHANGED from true to false — enforces SoundIn brand colors
    content: @Composable () -> Unit
) {
    val colorScheme = when {
        // Use dynamic color only if explicitly enabled AND on Android 12+
        dynamicColor && Build.VERSION.SDK_INT >= Build.VERSION_CODES.S -> {
            val context = LocalContext.current
            if (darkTheme) dynamicDarkColorScheme(context) else dynamicLightColorScheme(context)
        }
        darkTheme -> DarkColorScheme   // Our custom dark palette
        else -> LightColorScheme       // Our custom light palette (default)
    }
    MaterialTheme(
        colorScheme = colorScheme, // Applies brand colors to all composables inside
        typography = Typography,
        content = content
    )
}
Parameters explained
dynamicColor = falseDisables Material You's wallpaper-based color extraction. Setting it to false ensures SoundIn's purple brand identity is always visible, regardless of the user's wallpaper choice.
contentA composable lambda that receives all theme tokens. Every composable nested inside can call MaterialTheme.colorScheme.*, MaterialTheme.typography.*, etc.

3Login UI

Building the complete Login Screen with Scaffold, OutlinedTextField, Switch and Snackbar

In this chapter we build the complete Login Screen. The screen is split into two composables: LoginScreen manages the Scaffold, SnackbarHostState and LoginViewModel observations. LoginContent is purely visual — it receives all values as parameters, making it easy to Preview and test independently.

🎯 UX Decisions in this chapter
  • Email keyboard — KeyboardType.Email automatically shows the @ key on mobile keyboards
  • Password toggle — users can verify what they typed before submitting
  • isError + supportingText — the user sees exactly which field is wrong and why
  • Remember session Switch — the user controls whether the app saves their credentials
  • Snackbar with Dismiss — the user decides when the feedback disappears

Create the file

Create a new Kotlin file called LoginScreen.kt inside the ui/screens package. This file contains both LoginContent (written first) and LoginScreen (written below it), followed by the Preview at the bottom.

Build LoginContent

LoginContent is a stateless composable — it receives all values as parameters and has no state of its own except for the local password visibility toggle. This is the correct pattern in Compose: push state up, keep leaf composables dumb.

Place LoginContent at the top of LoginScreen.kt, immediately after the imports. It will be called by LoginScreen inside the Scaffold's content lambda.
ui/screens/LoginScreen.kt — LoginContent (complete)
@Composable
fun LoginContent(
    paddingValues: PaddingValues,         // Insets from Scaffold — prevents content hiding behind TopAppBar
    email: String,                         // Current email value observed from ViewModel
    onEmailChanged: (String) -> Unit,      // Called every time the user types in the email field
    emailError: Boolean,                   // true = show red border and error message below the field
    password: String,
    onPasswordChanged: (String) -> Unit,
    passwordError: Boolean,
    rememberSession: Boolean,              // Current Switch state observed from ViewModel
    onRememberSession: (Boolean) -> Unit,
    onLoginClick: () -> Unit,              // Triggered when the "Log In" button is tapped
    onNavigationToRegister: () -> Unit     // Triggered when the "Register" TextButton is tapped
) {
    // Column arranges all form elements vertically
    // verticalScroll allows scrolling on small screens or when the keyboard is open
    Column(
        modifier = Modifier
            .fillMaxSize()
            .background(MaterialTheme.colorScheme.background)
            .padding(paddingValues)             // Apply Scaffold insets first
            .verticalScroll(rememberScrollState())
            .padding(24.dp),                    // Then add visual padding around the content
        horizontalAlignment = Alignment.CenterHorizontally,
        verticalArrangement = Arrangement.spacedBy(16.dp) // 16dp gap between all children automatically
    ) {
        Spacer(modifier = Modifier.height(32.dp)) // Extra top space for visual breathing room

        // App logo — the "S" initial displayed in the primary brand color
        Text(
            text = "S",
            style = MaterialTheme.typography.displayLarge,
            color = MaterialTheme.colorScheme.primary
        )

        // App name
        Text(
            text = "SoundIn",
            style = MaterialTheme.typography.headlineMedium
        )

        Spacer(modifier = Modifier.height(16.dp))

        // ── EMAIL FIELD ──
        // KeyboardType.Email tells the OS to show the email keyboard (@ key visible)
        // isError = emailError draws a red border when the email format is invalid
        // supportingText shows the error message below the field — only visible when isError = true
        OutlinedTextField(
            value = email,
            onValueChange = { onEmailChanged(it) },
            label = { Text("Email Address") },
            isError = emailError,
            supportingText = {
                if (emailError) Text("Enter a valid email address")
            },
            keyboardOptions = KeyboardOptions(
                keyboardType = KeyboardType.Email, // Shows @ key on the mobile keyboard
                imeAction = ImeAction.Next          // "Next" button moves focus to the password field
            ),
            modifier = Modifier.fillMaxWidth()
        )

        // ── PASSWORD FIELD ──
        // passwordVisible is local state — it only controls the eye icon toggle
        // It does not need to survive rotation, so remember{} without ViewModel is appropriate here
        var passwordVisible by remember { mutableStateOf(false) }

        OutlinedTextField(
            value = password,
            onValueChange = { onPasswordChanged(it) },
            label = { Text("Password") },
            keyboardOptions = KeyboardOptions(
                keyboardType = KeyboardType.Password,
                imeAction = ImeAction.Done // Last field — "Done" closes the keyboard
            ),
            isError = passwordError,
            supportingText = {
                if (passwordError) Text("Minimum 8 characters")
            },
            // PasswordVisualTransformation replaces characters with dots
            // VisualTransformation.None shows the actual characters when toggle is active
            visualTransformation = if (passwordVisible) {
                VisualTransformation.None
            } else {
                PasswordVisualTransformation()
            },
            trailingIcon = {
                // Eye icon button — tapping it flips passwordVisible between true and false
                IconButton(onClick = { passwordVisible = !passwordVisible }) {
                    Icon(
                        imageVector = if (passwordVisible) {
                            Icons.Default.Visibility     // Eye open — password is visible
                        } else {
                            Icons.Default.VisibilityOff  // Eye closed — password is hidden
                        },
                        contentDescription = if (passwordVisible) "Hide password" else "Show password"
                    )
                }
            },
            modifier = Modifier.fillMaxWidth()
        )

        // ── REMEMBER SESSION SWITCH ──
        // Row places the label on the left and the Switch on the right
        // SpaceBetween pushes them to opposite ends of the available row width
        Row(
            modifier = Modifier.fillMaxWidth(),
            verticalAlignment = Alignment.CenterVertically,
            horizontalArrangement = Arrangement.SpaceBetween
        ) {
            Text(
                text = "Remember session",
                style = MaterialTheme.typography.bodyMedium
            )
            Switch(
                checked = rememberSession,
                onCheckedChange = { onRememberSession(it) }
            )
        }

        // ── LOGIN BUTTON ──
        // Primary action — full width following Material 3 button guidelines for prominent actions
        Button(
            onClick = { onLoginClick() },
            modifier = Modifier.fillMaxWidth()
        ) {
            Text("Log In")
        }

        // ── REGISTER LINK ──
        // TextButton has no background or border — it looks like a text hyperlink
        // Used for secondary actions that are less important than the primary button
        TextButton(onClick = onNavigationToRegister) {
            Text("Don't have an account? Register")
        }

    } // end of Column
}
Key concepts in LoginContent
paddingValuesReceived from the Scaffold's content lambda. Must be applied to the top-level composable so content is never hidden behind the TopAppBar or SnackbarHost.
Arrangement.spacedBy(16.dp)Adds 16dp of space between every child in the Column automatically — cleaner than adding Spacer composables between each field.
var passwordVisible by rememberLocal state for the visibility toggle. It does not need to survive rotation (a user would not expect the password to stay visible after rotating), so remember without ViewModel is appropriate.
ImeAction.Next / DoneControls the action button on the mobile keyboard. Next moves focus to the next field; Done closes the keyboard. These create a smooth form-filling experience on mobile.

Build LoginScreen

LoginScreen is the stateful composable. It creates the Scaffold, observes the LoginViewModel state flows, and handles the Snackbar logic inside a coroutine scope. Place it below LoginContent in the same file.

Place LoginScreen after the closing brace of LoginContent, in the same file LoginScreen.kt. It calls LoginContent inside the Scaffold's content lambda — the block that follows the Scaffold's closing parenthesis.
ui/screens/LoginScreen.kt — LoginScreen (complete)
@OptIn(ExperimentalMaterial3Api::class)
@Composable
fun LoginScreen(
    viewModel: LoginViewModel = viewModel(),      // Injected by Compose — survives rotation
    sessionViewModel: UserSessionViewModel,        // Manages the login session across the entire app
    onNavigationToRegister: () -> Unit,           // LoginScreen does not know where Register is — callback
    onLoginSuccess: () -> Unit                     // Called after successful login and Snackbar
) {
    // SnackbarHostState manages the queue and display of Snackbars
    val snackBarHostState = remember { SnackbarHostState() }

    // rememberCoroutineScope provides a scope tied to this composable's lifecycle
    // Required because showSnackbar() is a suspend function
    val scope = rememberCoroutineScope()

    // Observe each StateFlow from the ViewModel
    // collectAsStateWithLifecycle() is lifecycle-aware — stops collecting when screen is not visible
    val email by viewModel.email.collectAsStateWithLifecycle()
    val password by viewModel.password.collectAsStateWithLifecycle()
    val rememberSession by viewModel.rememberSession.collectAsStateWithLifecycle()
    val emailError by viewModel.emailError.collectAsStateWithLifecycle()
    val passwordError by viewModel.passwordError.collectAsStateWithLifecycle()

    Scaffold(
        topBar = {
            TopAppBar(
                title = { Text("SoundIn") },
                colors = TopAppBarDefaults.topAppBarColors(
                    containerColor = MaterialTheme.colorScheme.primary,      // Purple background
                    titleContentColor = MaterialTheme.colorScheme.onPrimary  // White text
                )
            )
        },
        snackbarHost = {
            // SnackbarHost connects the SnackbarHostState to the Scaffold's Snackbar slot
            // Snackbars launched via snackBarHostState.showSnackbar() appear at the bottom here
            SnackbarHost(hostState = snackBarHostState)
        }
    ) { paddingValues -> // paddingValues = space occupied by TopAppBar + SnackbarHost
        LoginContent(
            paddingValues = paddingValues,
            email = email,
            onEmailChanged = viewModel::onEmailChanged,   // Method reference — cleaner than a lambda
            emailError = emailError,
            password = password,
            onPasswordChanged = viewModel::onPasswordChanged,
            passwordError = passwordError,
            rememberSession = rememberSession,
            onRememberSession = viewModel::onRememberSessionChanged,
            onLoginClick = {
                // validateAndLogin() validates fields, updates error states, returns Boolean
                val isValid = viewModel.validateAndLogin()
                scope.launch {
                    if (isValid) {
                        // Store the session before navigating — data is ready when ProfileScreen reads it
                        sessionViewModel.login(
                            name = "John Doe", // Dummy name — in production this comes from the server
                            email = email      // Real email typed by the user
                        )
                        // showSnackbar() is suspend — the coroutine pauses here until the Snackbar finishes
                        // onLoginSuccess() only runs AFTER the user reads the welcome message
                        onLoginSuccess()
                    } else {
                        snackBarHostState.showSnackbar(
                            message = "Please review the marked fields"
                        )
                    }
                }
            },
            onNavigationToRegister = onNavigationToRegister
        )
    } // end of Scaffold
}

Key concepts in LoginScreen
rememberCoroutineScope()Creates a coroutine scope tied to this composable. When the composable leaves the screen, the scope is automatically cancelled — preventing memory leaks from dangling coroutines.
collectAsStateWithLifecycle()Converts a StateFlow into a Compose State. The WithLifecycle variant stops collecting when the screen is in the background, saving battery and CPU compared to collectAsState().
viewModel::onEmailChangedKotlin method reference — equivalent to { value -> viewModel.onEmailChanged(value) } but shorter. Works because onEmailChanged takes exactly one String parameter.
scope.launch { }Launches a coroutine. Required because showSnackbar() is a suspend function — it can only be called from inside a coroutine.
sessionViewModel.login()Stores session data before calling onLoginSuccess(). This ensures the data is ready before any screen that depends on the session (like ProfileScreen) tries to read it.

4LoginViewModel

Separating UI state from business logic using StateFlow and ViewModel

In this chapter we create LoginViewModel and move all state and validation logic out of the composable. The ViewModel survives device rotation because it is scoped to the Activity lifecycle — not to the composable. After this chapter the Login Screen will preserve its data when the user rotates the device.

ℹ️ The core pattern: MutableStateFlow → StateFlow

Every piece of state follows the same two-variable pattern:
private val _field = MutableStateFlow(defaultValue) — private, writable, only the ViewModel can change it.
val field: StateFlow = _field.asStateFlow() — public, read-only, the UI observes this.

Create LoginViewModel.kt

Create a new Kotlin file called LoginViewModel.kt directly inside the ui package. ViewModels are not screen files — they live at the same level as the screens folder, not inside it.

File location: ui/LoginViewModel.kt — create this file at the same level as the screens folder, NOT inside it. ViewModels belong in the ui package alongside the ViewModel for Register.
ui/LoginViewModel.kt (complete)
class LoginViewModel : ViewModel() {

    // ── PRIVATE STATE ──
    // MutableStateFlow holds the current value and notifies all observers when it changes
    // The underscore prefix (_) signals that this variable is private and mutable
    private val _email = MutableStateFlow("")
    private val _password = MutableStateFlow("")
    private val _rememberSession = MutableStateFlow(false)
    private val _emailError = MutableStateFlow(false)    // false = no error shown to the user
    private val _passwordError = MutableStateFlow(false)

    // ── PUBLIC STATE ──
    // asStateFlow() wraps _email in a read-only StateFlow
    // The UI can observe these but cannot call .value = on them directly
    val email: StateFlow<String> = _email.asStateFlow()
    val password: StateFlow<String> = _password.asStateFlow()
    val rememberSession: StateFlow<Boolean> = _rememberSession.asStateFlow()
    val emailError: StateFlow<Boolean> = _emailError.asStateFlow()
    val passwordError: StateFlow<Boolean> = _passwordError.asStateFlow()

    // ── UPDATE FUNCTIONS ──
    // Called by the UI when the user types — the ViewModel updates the state
    // Clearing the error immediately when the user types again is good UX:
    // the red border disappears as soon as the user starts correcting the input
    fun onEmailChanged(value: String) {
        _email.value = value
        _emailError.value = false // Clear error as soon as user starts correcting
    }

    fun onPasswordChanged(value: String) {
        _password.value = value
        _passwordError.value = false
    }

    fun onRememberSessionChanged(value: Boolean) {
        _rememberSession.value = value
    }

    // ── VALIDATION ──
    // Called when the user taps "Log In"
    // Updates both error flags and returns true only if BOTH fields are valid
    fun validateAndLogin(): Boolean {
        val isEmailValid = _email.value.contains("@") && _email.value.contains(".")
        val isPasswordValid = _password.value.length >= 8

        // true = error (field is invalid), false = no error (field is valid)
        _emailError.value = !isEmailValid
        _passwordError.value = !isPasswordValid

        return isEmailValid && isPasswordValid
    }
}
Key concepts in LoginViewModel
: ViewModel()Inheriting from ViewModel makes the class lifecycle-aware. Android keeps the ViewModel alive during configuration changes (rotation, theme changes) and only destroys it when the Activity is truly finished.
MutableStateFlow("")Creates a hot stream that always has a value. The string "" is the initial value — what the field contains before the user types anything.
asStateFlow()Wraps the MutableStateFlow in a read-only StateFlow. The UI can observe it but cannot modify it directly. This enforces unidirectional data flow.
_emailError.value = falseClears the error as soon as the user starts typing again. The red border disappears while the user corrects the input, providing immediate positive feedback.

  • Without the ViewModel, both fields would reset to empty strings on every rotation

5Register Screen

Advanced Material 3 — DatePickerDialog, ExposedDropdownMenuBox, BackHandler + AlertDialog

This is the most complex chapter of the course. The Register Screen introduces three advanced Material 3 components plus a complete RegisterViewModel with seven fields and comprehensive validation.

🎯 UX Decisions in this chapter
  • DatePickerDialog — visual calendar is faster and less error-prone than typing a date. Users cannot accidentally enter invalid dates like February 31st.
  • ExposedDropdownMenuBox — a predefined genre list eliminates spelling errors and ensures consistent data in the backend.
  • BackHandler + AlertDialog — if a user fills in 5 of 7 fields and accidentally presses back, they lose everything. The confirmation dialog prevents this — exactly like Gmail and WhatsApp.
  • Confirm password field — forces the user to type the password twice, catching typos before account creation.
  • Accepted terms Switch — required for GDPR and app store compliance.

Create RegisterViewModel.kt

Create RegisterViewModel.kt in the ui package alongside LoginViewModel.kt. This ViewModel has 14 state variables — one value and one error flag for each of the 7 form fields.

File location: ui/RegisterViewModel.kt — same folder as LoginViewModel.kt, inside the ui package but NOT inside the screens folder.
ui/RegisterViewModel.kt (complete)
class RegisterViewModel : ViewModel() {

    // ── PRIVATE STATE ──
    // Each form field has TWO MutableStateFlow variables:
    // 1. The field value (what the user typed or selected)
    // 2. The field error (whether to show the red border and error message)
    private val _name = MutableStateFlow("")
    private val _nameError = MutableStateFlow(false)
    private val _email = MutableStateFlow("")
    private val _emailError = MutableStateFlow(false)
    private val _password = MutableStateFlow("")
    private val _passwordError = MutableStateFlow(false)
    private val _passwordConfirm = MutableStateFlow("")
    private val _passwordConfirmError = MutableStateFlow(false)
    private val _birthDate = MutableStateFlow("") // Set by DatePickerDialog
    private val _birthDateError = MutableStateFlow(false)
    private val _genre = MutableStateFlow("") // Set by ExposedDropdownMenuBox
    private val _genreError = MutableStateFlow(false)
    private val _acceptedTerms = MutableStateFlow(false) // Set by Switch
    private val _acceptedTermsError = MutableStateFlow(false)

    // ── PUBLIC STATE ──
    // All 14 StateFlow variables exposed to the UI
    val name: StateFlow<String> = _name.asStateFlow()
    val nameError: StateFlow<Boolean> = _nameError.asStateFlow()
    val email: StateFlow<String> = _email.asStateFlow()
    val emailError: StateFlow<Boolean> = _emailError.asStateFlow()
    val password: StateFlow<String> = _password.asStateFlow()
    val passwordError: StateFlow<Boolean> = _passwordError.asStateFlow()
    val passwordConfirm: StateFlow<String> = _passwordConfirm.asStateFlow()
    val passwordConfirmError: StateFlow<Boolean> = _passwordConfirmError.asStateFlow()
    val birthDate: StateFlow<String> = _birthDate.asStateFlow()
    val birthDateError: StateFlow<Boolean> = _birthDateError.asStateFlow()
    val genre: StateFlow<String> = _genre.asStateFlow()
    val genreError: StateFlow<Boolean> = _genreError.asStateFlow()
    val acceptedTerms: StateFlow<Boolean> = _acceptedTerms.asStateFlow()
    val acceptedTermsError: StateFlow<Boolean> = _acceptedTermsError.asStateFlow()

    // ── UPDATE FUNCTIONS ──
    // Text fields clear their error when the user starts typing — same pattern as LoginViewModel
    fun onNameChanged(newName: String) { _name.value = newName; _nameError.value = false }
    fun onEmailChanged(newEmail: String) { _email.value = newEmail; _emailError.value = false }
    fun onPasswordChanged(newPassword: String) { _password.value = newPassword; _passwordError.value = false }
    fun onPasswordConfirmChanged(newConfirm: String) { _passwordConfirm.value = newConfirm; _passwordConfirmError.value = false }
    fun onBirthDateChanged(newDate: String) { _birthDate.value = newDate } // DatePicker always returns a valid date — no error to clear
    fun onGenreChanged(newGenre: String) { _genre.value = newGenre; _genreError.value = false }
    fun onAcceptedTermsChanged(newValue: Boolean) { _acceptedTerms.value = newValue }

    // ── VALIDATION ──
    // Called when "Create Account" is tapped
    // Validates all 7 fields simultaneously and returns true only if ALL are valid
    fun validateAndRegister(): Boolean {
        _nameError.value = _name.value.isBlank()
        _emailError.value = _email.value.isBlank() || !_email.value.contains("@") || !_email.value.contains(".")
        _passwordError.value = _password.value.isBlank() || _password.value.length < 8
        // Confirm password must match the password field exactly
        _passwordConfirmError.value = _passwordConfirm.value.isBlank() || _passwordConfirm.value != _password.value
        _birthDateError.value = _birthDate.value.isBlank()
        _genreError.value = _genre.value.isBlank()
        _acceptedTermsError.value = !_acceptedTerms.value

        // All error flags must be false AND terms must be accepted

DatePickerDialog explained
rememberDatePickerState()Creates and remembers the state object used internally by DatePicker. It stores the currently highlighted and selected date in milliseconds since the Unix epoch (January 1, 1970).
selectedDateMillisThe number of milliseconds since January 1, 1970 for the selected date. It is null when no date has been selected yet. The ?.let { } block only runs when a date is actually selected.

ui/navigation/SoundInNavGraph.kt (complete)
@Composable
fun SoundInNavGraph(
    navController: NavHostController,      // The NavController created in MainActivity — passed down
    sessionViewModel: UserSessionViewModel // Shared across Login and Profile — passed from MainActivity
) {
    NavHost(
        navController = navController,
        startDestination = SoundInRoutes.LOGIN // App always starts on the Login Screen
    ) {

        // ── LOGIN DESTINATION ──
        composable(SoundInRoutes.LOGIN) {
            LoginScreen(
                sessionViewModel = sessionViewModel,
                onNavigationToRegister = {
                    // Navigate to Register — adds Register to the back stack on top of Login
                    navController.navigate(SoundInRoutes.REGISTER)
                },
                onLoginSuccess = {
                    navController.navigate(SoundInRoutes.MAIN) {
                        // Remove Login from the back stack after successful login
                        // inclusive = true means Login itself is removed (not just screens above it)
                        popUpTo(SoundInRoutes.LOGIN) { inclusive = true }
                    }
                }
            )
        }

        // ── REGISTER DESTINATION ──
        composable(SoundInRoutes.REGISTER) {
            RegisterScreen(
                onNavigateToLogin = {
                    navController.navigate(SoundInRoutes.LOGIN) {
                        // Remove Register from the back stack when navigating back to Login
                        popUpTo(SoundInRoutes.LOGIN) { inclusive = true }
                    }
                }
            )
        }

        // ── MAIN DESTINATION ──
        composable(SoundInRoutes.MAIN) {
            MainScreen(
                sessionViewModel = sessionViewModel,
                onLogout = {
                    navController.navigate(SoundInRoutes.LOGIN) {
                        // Remove Main from the back stack on logout
                        // The user cannot press back and return to Main after logging out
                        popUpTo(SoundInRoutes.MAIN) { inclusive = true }
                    }
                }
            )
        }
    }
}
Navigation concepts explained
NavHostThe container that manages which destination is currently visible. It uses the NavController to track the back stack and renders the correct composable based on the current route.
startDestinationThe first screen shown when the app launches. Set to SoundInRoutes.LOGIN so unauthenticated users always land on the login screen.
composable(route)Registers a screen as a navigation destination. The composable inside the lambda is rendered when the NavHost navigates to that route.
popUpTo(route) { inclusive = true }Removes all screens from the back stack up to and including the specified route. After login, popUpTo(LOGIN) { inclusive = true } removes Login so pressing back on Main does not return to Login.
